# Semantic Resolution Graph — Phase 8 Baseline

Phase 8 of the build plan
(`crates/o11a-analyze/docs/build-plans/semantic-resolution-graph.md`)
adds a `mechanical-graph` variant to the `--semantic-linking-compare-all`
harness so the recall × precision delta of the graph-driven resolver
(Phases 6 + 7) can be measured against the pre-graph baseline. This
document describes the methodology, the harness output files, and how
to record measurements that subsequent phases (9, 10) will compare
against.

The numbers in the "Results" section are filled in by an operator
running the harness against a real audit. They are not generated by
this plan's tests — the harness depends on live LLM calls and
substantial Solidity input and is impractical to run from CI. The doc's
job is to make those operator runs reproducible and comparable.

---

## What changed in Phase 8

Before Phase 8 the harness exposed five variants:

```
mechanical, bm25-gap, bm25-top-k-floor, bm25-permissive, llm
```

After Phase 8 the variant list is:

```
mechanical, mechanical-graph, bm25-gap, bm25-top-k-floor, bm25-permissive, llm
```

The semantic of `mechanical` also changed. Pre-Phase-6/7, `mechanical`
read the analyzer's `referenced_topic` directly off doc-tree
`CodeIdentifier` nodes — which at that point only carried Phase A
(name-index) resolutions, since the graph resolver did not yet exist.
Post-Phase-6/7 the analyzer overwrites those same fields with Phase B
(graph-driven) resolutions, so a naive `mechanical` would be
indistinguishable from `mechanical-graph`.

To preserve `mechanical` as the pre-graph baseline, the harness now
re-derives Phase-A-only mechanical links by passing the set of
Phase-B-resolved doc-node IDs (read from `audit_data.resolution_traces`)
to `context::mechanical_semantic_links_excluding`. The `mechanical-graph`
variant uses the unmodified `mechanical_semantic_links` — i.e. the
production state.

| Variant            | Resolver phases used   |
|--------------------|------------------------|
| `mechanical`       | Phase A only           |
| `mechanical-graph` | Phase A + Phase B      |
| `bm25-*`           | Phase A + Phase B + BM25 expansion |
| `llm`              | Phase A + Phase B + LLM Pass 1 / 2 |

BM25 / LLM variants seed from the post-Phase-B production baseline so
those workflows benchmark only their own additive contribution beyond
the graph resolver.

## How to run the harness

```
cargo run -p o11a-analyze --release -- analyze \
    --semantic-linking-compare-all \
    <project_root> <audit_id>
```

The harness writes its outputs to
`<project_root>/o11a/semantic-linking-compare/`. The recommended fixture
is `nudgexyz` (the largest curated audit in the repo at the time of
writing). LLM variants require model API credentials to be configured
per the project's standard setup.

### Output files relevant to Phase 8

| File                                  | Purpose                                                  |
|---------------------------------------|----------------------------------------------------------|
| `mechanical.jsonl`                    | Pass 2 candidates from Phase A only                      |
| `mechanical-graph.jsonl`              | Pass 2 candidates from Phase A + Phase B (new)           |
| `mechanical.pass3.jsonl`              | Pass 3 results for `mechanical`                          |
| `mechanical-graph.pass3.jsonl`        | Pass 3 results for `mechanical-graph` (new)              |
| `pass3-summary.jsonl`                 | Side-by-side per `(section, declaration)` across variants |
| `edge-contribution-histogram.jsonl`   | Per-`EdgeType` aggregate of Phase B's top contributing edges (new) |
| `mechanical-pass1.jsonl`              | Phase-A-only contract anchors (semantic now Phase A only) |
| `bm25-pass1-ranking.jsonl`            | Per `(section, contract)` BM25 score; `is_mechanical_anchor` reflects the production baseline |

Two runs of the harness against an unchanged audit must produce
byte-identical files. Phase 8 added a determinism contract test for the
harness pieces it touched — namely the histogram aggregator and the
`VARIANTS` ordering. Determinism for the harness as a whole is
established by the existing infrastructure.

## Methodology — the three signals to record

### 1. Recall delta

Number of new `(section, declaration)` pairs `mechanical-graph`
proposes that `mechanical` does not. Compute by diffing Pass 2 output:

```
cd <project_root>/o11a/semantic-linking-compare
jq -r '.section_topic + " " + .member_topic' mechanical.jsonl       | sort -u > /tmp/mech.pairs
jq -r '.section_topic + " " + .member_topic' mechanical-graph.jsonl | sort -u > /tmp/mech-graph.pairs
comm -13 /tmp/mech.pairs /tmp/mech-graph.pairs | wc -l   # pairs only mechanical-graph proposed
comm -23 /tmp/mech.pairs /tmp/mech-graph.pairs | wc -l   # pairs only mechanical proposed (should be 0)
```

The "only mechanical proposed" count is the regression check: Phase B
must never **drop** a Phase A resolution. A non-zero value here is a
bug in Phase 6/7, not a measurement of the graph.

### 2. Precision

Of the `mechanical-graph`-only pairs, what fraction survive Pass 3?
Look up each pair in `mechanical-graph.pass3.jsonl`:

```
# Pass 3 surviving pairs:
jq -r '.section_topic + " " + .declaration_topic' mechanical-graph.pass3.jsonl | sort -u > /tmp/mech-graph.pass3.pairs
# Intersection of new pairs and surviving pairs:
comm -12 /tmp/mech-graph.pass3.pairs <(comm -13 /tmp/mech.pairs /tmp/mech-graph.pairs) | wc -l
```

Divide by the recall-delta count from step 1 to get the precision
fraction.

### 3. Edge-contribution histogram

`edge-contribution-histogram.jsonl` reports, per `EdgeType`:

- `occurrences` — how often this edge type appeared in any
  resolution's top-three contributing edges
- `total_weighted_contribution` — sum of PR mass attributed to this
  edge type
- `traces_with_edge` — distinct resolutions that listed it
- `by_phase` — occurrence counts split by `ResolutionPhase`

This is the calibration signal. An edge type that never appears in any
top-three across a real audit is either redundant or
over/under-weighted; conversely, an edge type that dominates many
resolutions but contributes low weight per occurrence may need a higher
default weight.

## Beyond Phase 8

Phases 9 and 10 will add:

- Phase C (co-location pinning) and Phase D (re-iteration) to both
  resolution passes.
- Phase E (anchor-by-name fallback) plus `referenced_topic_candidates`
  population.

Each of those phases will re-run this harness and append a new
`Run N — …` section above. The diff between successive runs is the
phase's actual contribution.
